## Context

这是一个全新的 AI 驱动修仙模拟游戏项目。项目从零开始构建，需要创建完整的游戏架构。

**技术栈：**
- Python 3.13+
- pygame (UI 框架)
- uv (包管理器)
- 大语言模型 API (用于角色行为模拟)

**约束条件：**
- AI 上下文窗口有限，需要智能检索相关信息
- 游戏以数值为核心，属性变化必须精确计算
- 支持单角色和多角色交互两种场景

## Goals / Non-Goals

**Goals:**
- 实现完整的角色属性和物品系统
- 实现 AI 交互流程（提示词构建 → AI 响应 → 属性更新）
- 实现境界突破机制（小境界和大境界）
- 实现多角色探测和同步交互
- 实现战斗伤害计算系统

**Non-Goals:**
- 图形化 3D 渲染（仅 pygame 纯文字/2D UI）
- 多人在线功能
- 移动设备支持

## Decisions

### 1. 项目结构
**决策：** 采用模块化包结构，按功能域组织代码

```
src/
├── character/      # 角色系统
├── item/           # 物品系统
├── ai/             # AI 交互
├── combat/         # 战斗系统
├── realm/          # 境界系统
├── world/          # 世界管理
├── ui/             # UI 渲染
└── main.py         # 主入口
```

**理由：** 高内聚低耦合，便于后续扩展和维护。

### 2. 数据存储
**决策：** 使用 YAML 配置文件 + SQLite 数据库

- YAML：世界观配置、境界参数、基础法宝/功法模板
- SQLite：动态创建的物品、角色记忆、游戏状态

**理由：** YAML 易于人工编辑，SQLite 提供高效的数据检索能力。

### 3. AI 提示词策略
**决策：** 关键词检索 + 信息压缩

- 根据角色当前状态提取相关关键词
- 仅加载与当前场景相关的法宝、功法信息
- 记忆采用摘要方式，优先加载近期记忆

**备选方案：** 使用向量数据库进行语义检索
→ 暂不采用，增加复杂度且项目初期不需要

### 4. 游戏循环架构
**决策：** 主循环 + 事件驱动

```
主循环:
  1. 更新世界时间
  2. 检测角色交互
  3. 为每个角色/交互组请求 AI 决策
  4. 应用属性变化
  5. 检查境界突破
  6. 渲染 UI
```

**理由：** 清晰的流程控制，便于调试和扩展。

### 5. 不可变性设计
**决策：** 状态更新采用不可变模式

每次状态变化创建新的对象副本，而非修改原对象。

**理由：** 便于回滚、调试，避免副作用。

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AI 响应格式不稳定 | 高 | 严格的输出格式校验 + 重试机制 |
| AI API 成本过高 | 中 | 缓存常用场景、批量处理决策 |
| 多角色同步复杂 | 中 | 明确角色分组策略，限制每组最大人数 |
| 游戏平衡难以调整 | 中 | 所有数值参数配置化，便于调优 |
